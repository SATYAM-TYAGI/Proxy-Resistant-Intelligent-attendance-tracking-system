Proxy-Resistant Attendance Tracking System (PRATS)
Detailed Interview Explanation + Future DevOps Roadmap

========================================================================
1) Elevator Pitch (30-60 seconds)
========================================================================

This project is a full-stack, AI-assisted attendance platform designed to
reduce proxy attendance. Students and faculty register with face data
(selfie + short video), faculty run class sessions, and students mark
attendance through live camera frame bursts. A Python AI microservice
handles face embedding and liveness checks, while a Node.js backend owns
authentication, authorization, business rules, and persistence in MongoDB.

The key differentiator is that attendance is not just "face matched once":
it is duration-aware with a 70% rule, class session lifecycle
(start/end/leave), and faculty override controls.


========================================================================
2) Business Problem and Why This Matters
========================================================================

- Traditional attendance systems are easy to game (roll-call proxy, buddy
  check-ins, single snapshot loopholes).
- Pure manual attendance is slow and error-prone for faculty.
- A scalable solution needs role-based access, reliable records, and
  practical anti-spoofing measures.

This project addresses those with:
- Face identity verification
- Basic liveness heuristics
- Session-based duration tracking
- Faculty monitoring and manual exception handling


========================================================================
3) High-Level Architecture
========================================================================

Core services:
1. Frontend: React + Vite (student and faculty portals)
2. Backend API: Node.js + Express (JWT auth, business logic, DB orchestration)
3. AI Microservice: FastAPI + InsightFace + OpenCV (embedding + liveness)
4. Database: MongoDB + Mongoose models

Data and control flow:
- Browser calls backend endpoints under /api/*
- Backend validates user/session/role, fetches DB data, then calls AI service
  for image/video inference tasks
- AI service returns match/liveness results
- Backend writes final attendance state and metrics to MongoDB

Design benefit:
- AI is separated from business logic, so each layer scales independently.
- Backend remains source of truth; AI service is stateless for user identity.


========================================================================
4) Technology Stack and Why Each Was Chosen
========================================================================

Frontend (React + Vite):
- React is ideal for role-based dashboard flows and guarded routes.
- Vite provides quick development and optimized production builds.
- Camera access in browser enables no-install attendance marking.

Backend (Node.js + Express):
- Handles route orchestration, validation, auth, and role checks.
- Easy integration with multipart upload forwarding and JWT middleware.
- Keeps all attendance rules centralized.

Database (MongoDB + Mongoose):
- Flexible schema for evolving AI metadata.
- Natural fit for attendance documents and course/session relationships.
- Mongoose provides validation and query ergonomics.

AI Layer (Python FastAPI + InsightFace + OpenCV):
- Python ecosystem is stronger for CV/ML inference.
- FastAPI gives simple, high-performance inference APIs.
- InsightFace produces robust embeddings.
- OpenCV heuristics provide lightweight liveness checks.


========================================================================
5) Domain Model (What Is Stored)
========================================================================

User:
- role: student or faculty
- identity fields: sapId/facultyId, email, name
- security: passwordHash (bcrypt)
- biometrics: faceEmbedding (primary) and optional faceGallery

Course:
- name, code, faculty owner
- enrolled studentIds

ClassSession:
- courseId + facultyId
- date, startTime, endTime
- status: active/inactive

Attendance:
- studentId + classSessionId
- entryTime and optional exitTime
- plannedDurationMs, attendedDurationMs, attendancePercentage
- status: pending/present/absent
- manualOverride flag for faculty interventions


========================================================================
6) End-to-End User Flows
========================================================================

A) Registration Flow (Student/Faculty)
- User submits details plus selfie and short video.
- Backend validates required fields and duplicate accounts.
- Backend forwards media to AI /extract-registration.
- AI extracts embeddings from selfie + sampled video frames.
- AI returns primary and gallery embeddings.
- Backend stores user record and issues JWT.

B) Login Flow
- User submits email/password.
- Backend verifies hash using bcrypt.
- JWT issued for stateless authorization.

C) Faculty Session Management
- Faculty creates course.
- Faculty schedules class session.
- Faculty starts class to activate attendance window.
- Faculty ends class; backend finalizes metrics/status.

D) Student Attendance Flow
- Student enrolls in a course.
- Student sees active classes.
- On mark attendance:
  - Browser captures multiple JPEG frames over a short period.
  - Backend validates class is active and student is enrolled.
  - Backend sends frames + enrolled candidate embeddings to AI.
  - AI runs liveness + identity match and returns result.
  - Backend verifies matched sap_id equals logged-in student.
  - Backend creates/updates attendance row and computes metrics.

E) Leave and Finalization
- Student can call "leave" to set exitTime.
- While class active, attendance percentage updates progressively.
- After class ends, backend finalizes present/absent by threshold.

F) Faculty Manual Override
- Faculty can force present/absent for exceptional scenarios.
- manualOverride prevents AI-based recalculation from overriding intent.


========================================================================
7) Anti-Proxy Logic and Security Controls
========================================================================

1. Role-Based Access Control
- Student-only and faculty-only routes enforce workflow boundaries.

2. JWT Authentication
- Protects all sensitive endpoints.
- User context (id + role) drives authorization checks.

3. Enrollment and Session Validation
- Student cannot mark attendance unless enrolled and class is active.

4. Identity Mismatch Guard
- Even if AI returns another student as best match, backend cross-checks
  against logged-in student's SAP ID and rejects mismatch.

5. Liveness Checks
- AI verifies motion/sharpness/frame changes to reduce spoof attempts.
- Not perfect against advanced attacks, but stronger than static-photo-only.

6. Duration-Based Rule
- 70% attendance threshold discourages "quick in-and-out" proxy behavior.


========================================================================
8) Attendance Calculation Logic (Interview-Friendly)
========================================================================

Key policy:
- Attendance status depends on effective presence duration, not only entry.

Computation:
- plannedDurationMs = scheduledEnd - scheduledStart
- attendedDurationMs = effectiveEnd - effectiveStart
- attendancePercentage = attended/planned * 100
- present if percentage >= 70, else absent

Notes:
- During active class without exitTime, effectiveEnd uses min(now, endTime)
  so progress is visible.
- On class end, missing exitTime is treated by policy (session-end baseline).
- Faculty manual override can force final status.

10) Future DevOps Roadmap (Jenkins, Kubernetes, etc.) now done 
========================================================================

Phase 1: CI Foundation (Jenkins + Quality Gates)
- Create Jenkins pipeline with stages:
  1) Checkout
  2) Install dependencies (client/server/ai-service)
  3) Lint + static checks
  4) Unit/integration tests
  5) Build artifacts/images
  6) Security scans
- Enforce "pipeline must pass to merge" policy.
- Publish test reports and code coverage in Jenkins.

Phase 2: Container Registry + Image Strategy
- Build immutable versioned Docker images per service.
- Push to registry (Docker Hub, ECR, GCR, or ACR).
- Tag strategy:
  - semantic tags for releases (v1.2.0)
  - commit SHA tags for traceability

Phase 3: Kubernetes Orchestration
- Move compose services to Kubernetes manifests or Helm charts:
  - Deployments: client, server, ai-service
  - StatefulSet: MongoDB (or use managed DB service)
  - Services: ClusterIP for internal, Ingress for external
- Add readiness/liveness probes:
  - server: /api/health
  - ai-service: /health
- Use Horizontal Pod Autoscaler:
  - server scale by CPU/RPS
  - ai-service scale by CPU/GPU utilization or queue depth

Phase 4: Progressive Delivery
- Use rolling updates with maxUnavailable and maxSurge controls.
- Add blue/green or canary for backend and AI model changes.
- Keep rollback strategy:
  - image tag rollback
  - config rollback via GitOps history

Phase 5: GitOps and Environment Promotion
- Keep K8s manifests/Helm values in Git.
- Use Argo CD or Flux for continuous reconciliation.
- Promotion model:
  - dev -> staging -> production
  - manual approval gate before production

Phase 6: Observability and SRE Practices
- Logs: centralized with ELK/Loki.
- Metrics: Prometheus + Grafana dashboards.
- Tracing: OpenTelemetry + Jaeger/Tempo for request path
  (browser -> backend -> AI).
- Alerting:
  - attendance API error rate
  - AI latency and failure ratio
  - DB connection saturation
  - auth anomaly spikes

Phase 7: Security and Compliance Hardening
- Store secrets in Kubernetes Secrets + external vault integration.
- Enforce TLS everywhere and secure Ingress.
- Add rate limiting and WAF for public endpoints.
- Run dependency/container scans (Snyk/Trivy/OWASP checks).
- Use least-privilege service accounts and network policies.
- Add audit trails for attendance modifications/manual overrides.

Phase 8: Reliability and Data Protection
- Configure MongoDB backup/restore drills.
- Multi-zone deployment for high availability.
- Define SLOs:
  - API uptime target
  - max attendance mark latency
  - AI success rate thresholds
- Add chaos/resilience testing to validate fault tolerance.

========================================================================
12) Known Limitations and Honest Improvements
========================================================================

Current limitations:
- Liveness is heuristic, not enterprise-grade anti-spoof.
- Class-size growth can increase candidate matching cost.
- JWT token revocation is basic in stateless mode.
- Course enrollment in demo mode is simpler than strict enterprise flow.

Planned improvements:
- Add stronger anti-spoof model + challenge-response prompts.
- Add vector index (FAISS) for large candidate pools.
- Implement refresh tokens + token revocation/rotation.
- Add stricter enrollment approvals and admin workflow.
- Add automated end-to-end test suites for camera simulation.
